# Cadence 项目规则

## 1. 工作目录边界

禁止修改当前工作目录（打开会话的目录）外的文件，除非获得用户明确同意。

## 2. 注释要求

新增接口和函数代码时，要加清晰注释；注释应解释对外契约、边界和非显然决策，不写空泛复述。

## 3. 简洁优先

用解决问题所需的最少代码。不添加超出要求之外的功能，不为只用一次的代码做抽象，不加入未被要求的灵活性或可配置性。

## 4. 精准外科式修改

只动必须动的地方。不要顺手优化相邻代码、注释或格式；不要重构没有坏掉的东西；与既有风格保持一致。移除因本次改动产生的未使用 import、变量、函数，但不要删除无关既有死代码，除非用户要求。

检验标准：每一行被改动的代码都应能直接追溯到用户请求。

## 5. Cadence 设计与适配规则

以下规则用于维护 cadence 仓库自身，修改 skill、agent、command、文档或跨宿主适配时同样遵守。

1. 不要低估模型能力：设计 skill / agent / slash command 时，模型靠自身能力就能做到的别写进提示词，只保留模型做不到或易做错的硬约束。提示词能省则省，不反复啰嗦。
2. 主对话 agent 只能看到子 agent 的 description，看不到正文。凡「主对话要正确调用就必须知道」的信息（输入参数、触发条件等），都要精简写进 description。
3. 子agent的description要在一段话里讲清四件事：①一句话定位（这个agent做什么）；②主动触发条件，用强指令写明"何时必须优先调用本agent而不是主agent直接动手"；③输入参数（参数名+含义，标注必填/可选）；④边界——何时不该用，并指向替代方案。触发条件中英双语各写一遍，便于主agent匹配。
4. Codex 与 Claude Code 适配一致：cadence 同时适配 Codex 和 Claude Code；除宿主工作模式、slash command 支持、工具命名等必要差异外，流程逻辑、判断标准和产物要求应保持一致。每次修改涉及任一端时，都要检查另一端是否需要同步。
5. 整体迭代：修改 cadence 文档、skill、agent 或 command 时，先通读相关整体语境，再决定改动；只保留会改变模型行为的约束，删掉无关规则和冗余话术。例如改 pai.md 后，要检查是否引入与 PAI 无关的规则或啰嗦内容。
