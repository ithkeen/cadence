# 测试驱动开发

## 理念

一等公民：
测试用例测的是功能，而不是功能实现的细节。一套测试用例就是一套预期行为，代码重构时，测试用例仍能保留。

设计测试用例时，遵守如下规则：
只有API接口、对外的方法和通用函数才值得写测试用例，其他的内部实现逻辑、内部使用函数和框架代码不需要写测试用例， 测试用例必须少而精。
测试用例是否能够测一种功能的场景，一套测试用例是否能够测一套预期行为。

**核心原则**：测试应当透过公共接口去验证行为，而不是验证实现细节。代码可以推倒重写，测试不该跟着动。

**好测试**是集成风格的：透过公共 API 走真实的代码路径，描述的是系统*做什么*，而不是*怎么做*。一个好测试读起来像一份规约——"用户能用有效购物车结账"准确告诉你存在什么能力。这类测试在重构中能活下来，因为它们不关心内部结构。

**坏测试**则与实现耦合：mock 内部协作者、测试私有方法、或者绕过接口走外部手段去验证（比如直接查数据库而不是走接口）。预警信号是：你重构时测试挂了，但行为没变。如果只是改了一个内部函数的名字测试就挂，那这些测试测的是实现，不是行为。

示例见 [tests.md](tests.md)，mock 指南见 [mocking.md](mocking.md)。

## 正确做法：垂直切片

使用"曳光弹"做垂直切片：一个测试 → 一个实现 → 重复。每个测试都是对上一轮所学的回应。代码刚刚是你自己写的，所以你准确知道哪些行为重要、该怎么验证。

```
RED→GREEN： test1→impl1
RED→GREEN： test2→impl2
RED→GREEN： test3→impl3
...
```

## 工作流

### 1. 规划

探索代码库时，使用项目的领域术语表，让测试名和接口词汇与项目本身的语言保持一致；同时尊重你所触及区域的 ADR。

动手写代码之前，下面这份**契约**必须就位：

- [ ] 公共接口形状（要新增 / 修改的方法签名）
- [ ] 要测的行为清单（不是实现步骤），按优先级排好，标出哪些是关键路径
- [ ] 按[可测试性](interface-design.md)设计接口

**这份契约怎么拿到，取决于你的运行模式：**

- **作为主对话 agent 运行时**：从上下文无法拿到完整契约时，直接用对话询问用户
- **作为子 agent 被派发**：从派发 prompt / 可读额外上下文文件 里获取不到完整契约时，直接报错

**禁止**在契约不完整的情况下开始写测试。

**你不可能把所有东西都测了。** 把测试精力集中在关键路径和复杂逻辑上，不要把每一个可能的边界情况都覆盖

### 2. 曳光弹

只写一个能验证系统一件事的测试：

```
RED：   为第一个行为写测试 → 测试失败
GREEN： 写最少的代码让它通过 → 测试通过
```

这就是你的曳光弹——证明这条路能端到端跑通。

### 3. 增量循环

对剩下的每一个行为：

```
RED：   写下一个测试 → 失败
GREEN： 写最少的代码让它过 → 通过
```

规则：

- 一次一个测试
- 只写够让当前测试通过的代码
- 不要预判后续测试
- 测试只盯可观察的行为

### 4. 重构

所有测试通过后，找[重构候选](refactoring.md)：

- [ ] 抽出重复代码
- [ ] 在自然的地方应用 SOLID 原则
- [ ] 想一想新代码让你对既有代码有了什么新认识
- [ ] 每一步重构之后都跑一遍测试

**红灯状态下绝不重构。** 先回到绿灯再说。

## 每轮循环的检查清单

```
[ ] 测试描述的是行为，不是实现
[ ] 测试只走公共接口
[ ] 内部重构后测试仍能存活
[ ] 这一轮写出来的代码是最小化的
[ ] 没有加入投机性的功能
```
