# 这个命令做什么

需求已由 `/cadence:pai` 锁定。本命令**不重做需求分析**，先消费 pai 文档里每个 `REQ` 内的 `相关上下文`，核验相关的现有入口、接口、schema、调用方、测试和约束，再回答一个问题：每个 `REQ` 技术上怎么实现？——技术栈、模块怎么拆、接口和数据怎么定、函数 / 方法契约是什么、每条 `AC` 怎么验证，全部钉到下游拿着文档就能直接做实施计划、不必返工。

产出 `may-<主题>.md` 是独立技术设计文档，只引用 pai 的路径、需求编号、标题、必要摘要和 AC 编号；不复制 pai 正文。源 pai 文档不动。

# 全局约束

- 全程中文。主 agent 不直接读项目源码——需要了解现有代码时调起 Explore 子 agent（见下），侦察结果不向用户复述。未完成上下文门禁前，不进入技术设计。
- 不拆任务、不写实现代码/伪代码，不改写需求（属 pai）。可以写函数 / 方法名、入参、返回值和错误语义；这是技术设计的对外契约，不属于实施代码。
- 决策点用 `AskUserQuestion`，每轮最多 1~2 题。**选型类决策（技术栈/模块拆法/算法/持久化等）先用一次 AskUserQuestion 摆 2-3 个备选+推荐+理由让用户选定方向**，再就该方向追问细节——别把方案对比和细节追问揉进同一问。
- 用户说"别问了直接落档"，设计没聊透就继续问。
- 引用了 research 的关键事实（具体值/字段名/endpoint/限流参数/协议字段）必须 inline 到设计段对应小节，下游默认只读这份文档。

# 流程

## 0. 解析参数 + 定位

`$ARGUMENTS` 是 pai 需求文档路径。缺失则报错退出：

```
❌ 缺少 pai 需求文档路径。用法：/cadence:may <pai-需求.md 路径>
请先跑 /cadence:pai 产出需求文档，再把它的路径传进来。
```

去空白得 `<pai-path>`。**主题**：取文件名去 `.md`，若以 `pai-` 开头则剥掉前缀。**产物路径**：`<pai 同目录>/may-<主题>.md`（即 `.cadence/cycle-<主题>/may-<主题>.md`）。

产物已存在 → AskUserQuestion 问「覆盖重做」还是「退出」。

## 1. 加载

Read `<pai-path>` 全文，作为**既定需求**通读（每个 `REQ` 要做什么、给谁、每条 `AC` 怎么验收、`相关上下文` 要求核验什么）——不重构、不改写，落档时只引用必要摘要，不复制正文。pai 正文即结论；`.cadence/research/` 是过程材料，此处不全量读，按需定点取用。

## 2. 上下文门禁

先从 pai 文档提取每个 `REQ` 内的 `相关上下文`，建立核验清单：

- 现状定位：已知的接口 / 命令 / 页面 / 组件 / 数据对象 / 外部服务。
- 已知事实：pai 阶段已经确认的现状。
- 设计前必查：设计前必须核验的入口、schema、调用方、测试、配置、兼容性约束等。
- 待核验问题：会影响技术设计但尚未确认的事实。

需要了解现有代码（模块结构、接口形态、实现位置）时调起 Explore：

```
Agent(subagent_type="Explore", description="<一句话定位>",
      prompt="<找什么 / 哪些路径 / 期望产出。广度：quick|medium|very thorough>")
```

门禁规则：

- `REQ` 有 `相关上下文` 时，逐条完成其中的 `设计前必查` 和 `待核验问题`；查不到就问用户澄清，不能跳过后直接写方案。
- `REQ` 缺少 `相关上下文`，或某个涉及现有系统的 `REQ` 只有泛泛描述（例如“修改一个接口”但没有接口名、路径、命令、页面或数据对象），先用 Explore 做 quick 级定位；仍定位不到就问用户补充具体对象。
- 全新功能可以记录“无现有对象”，但仍要核验项目入口约定、技术栈、测试框架和运行命令。
- 每个 `REQ` 至少形成一组可写入设计段的“现状依据”：相关上下文、已核验事实、仍需在实施中保持不变的规格。

## 3. 设计追问循环

如果追问中出现门禁阶段未覆盖的新代码事实缺口，继续调起 Explore 做窄范围补查；补查结论并入对应 `REQ` 的现状依据。

**每轮**：① 调研扫描（见下）→ ② 追问 1~2 题 → ③ 判断是否聊透。
某个 `REQ` 模糊到无法据此做技术决策 → 直接问用户澄清（不改写 pai 文档），结果体现进该 `REQ` 的设计段。

**覆盖维度**（按需求性质挑相关的问，不必逐项）：技术选型（**含测试框架与运行命令，必问到具名**）、每个 `REQ` 的模块划分、数据模型、接口设计、函数 / 方法契约、错误/失败语义、关键流程、`AC` 验证方式、非功能约束（性能/安全/可观测/部署）、风险与不确定项（**每项现场拍板，不悬置到实施**）。

**含 UI/页面/组件产出**才追加视觉维度（纯后端/库/CLI 跳过）：
- 每块 UI 产出逐块定 **mode**：`greenfield`（新页面/独立模块，可放飞）或 `inherit`（在既有页面上改造，须对齐既有视觉），两种可共存。
- `greenfield` 块定 **aesthetic_direction**，必须从这 12 个里选一个（拒绝"现代简洁"这类空话，选不出就 AskUserQuestion 摆方案）：`brutalist` / `editorial` / `luxury-refined` / `playful` / `retro-futurist` / `industrial` / `soft-pastel` / `art-deco` / `maximalist-chaos` / `brutally-minimal` / `cyberpunk` / `organic-natural`；有参考 URL/截图记为 reference_urls。
- `inherit` 块定对齐锚点（哪个页面/组件/design token）。
- 公用：色板（hex）、字体与排版、组件库选型（带版本）、暗色模式、关键交互动效。

**聊透的标准**（内部自检，不输出过程）：pai 每个 `REQ` 都有独立实现映射；每个 `REQ` 都有来自门禁阶段的现状依据；每条 `AC` 都能映射到测试或可观察行为；每个新增/改动模块边界清晰（不读内部能懂它做什么、改内部不破坏消费者）；函数 / 方法契约足够让下游知道入参、返回值和错误语义；不确定项全部拍板；选型理由已记入决策清单；含 UI 则各块 mode/aesthetic/锚点/视觉规范均已定。把自己当下游：拿这份设计能直接产出实施计划吗？没聊透就回到追问。

聊透后 AskUserQuestion 问「直接落档」还是「再聊」。

## 4. 落档

Write `<pai 同目录>/may-<主题>.md`，写成独立技术设计文档，一次写入：

```markdown
# 技术设计：<主题>

## 需求来源
- PAI 文档：`<pai-path>`
- 设计范围：REQ-001、REQ-002

## 技术栈
- 语言 / 框架 / 关键依赖（带版本）
- 测试框架与运行命令

## 需求实现映射

### REQ-001：<需求点标题>
- 需求摘要：<从 pai 提炼的一句话摘要，只用于定位，不改写规格>
- 关联验收标准：AC-001、AC-002
- 实现结论：<这条需求怎么落地，一句话讲清>

#### 现状依据
- 相关上下文：<来自 pai 的现状定位和已知事实>
- 已核验事实：<may 阶段核验到的入口、模块、schema、调用方、测试、配置等>
- 规格约束：<实施时必须保持不变的接口字段、错误语义、兼容性或用户可见行为；没有则写“无”>

#### 模块划分
- <模块 / 文件 / 包>：新增 / 修改 / 沿用；<职责>

#### 数据模型
- <实体名>：
  - 字段：<字段名、类型、含义、边界>
  - 关系：<与其他实体关系>
  - 持久化方式：<数据库 / 文件 / 内存 / 外部服务>

#### 接口设计
- <接口名>：
  - 方法 / 协议：<HTTP / CLI / 内部调用等>
  - 路径 / 命令 / 入口：<具体入口>
  - 入参：<参数名、类型、必填、边界>
  - 出参：<字段名、类型、含义>
  - 错误情形：<状态码 / 错误码 / 错误结构>

#### 函数 / 方法契约
- `<functionName>(<params>): <returnType>`
  - 职责：<做什么>
  - 入参边界：<接受什么、不接受什么>
  - 返回语义：<成功返回什么，失败怎么表达>
  - 副作用：<读写 DB / 文件 / 网络 / 缓存等>
  - 调用方：<谁调用它>

#### 关键流程
1. <步骤>
2. <步骤>

#### 错误处理
- <错误场景>：<错误结构 / 错误码 / 可重试性 / 降级策略>

#### 验收标准映射
- AC-001 -> <通过什么测试或行为验证>
- AC-002 -> <通过什么测试或行为验证>

### REQ-002：<需求点标题>
- ...

## 跨需求设计
- 共享模块：<多个 REQ 共用的模块>
- 共享数据结构：<DTO / schema / 类型>
- 共享错误模型：<统一错误码、异常结构>
- 共享测试策略：<公共 fixture、mock、测试数据>

## 非功能性约束
- 性能 / 安全 / 可观测 / 部署

## 视觉与交互风格
<!-- 仅 UI 需求写本节，纯后端 / 库 / CLI 整节删掉 -->
### UI 产出块清单
- <块名>：mode = greenfield | inherit
### 各块视觉锚点
- **<块名>（greenfield）**：aesthetic_direction = <12 枚举之一>；reference_urls = <URL 或"无">
- **<块名>（inherit）**：对齐锚点 = <既有页面 / 组件库 / design token>
### 公用视觉规范
- 色板：主色 <hex> / 强调色 <hex> / 中性色 <hex>
- 排版：字体族 / 字号尺度
- 组件库：<选型与版本>
- 暗色模式：是 / 否
- 关键交互：<hover / 过渡 / 加载态 / 微动效>

## 决策清单
- 选 X 不选 Y，因为 ...

## 实施边界
- 必须实现：REQ-001、REQ-002
- 不得实现：<防止 code-executor 顺手加功能>
- 不得变更的规格：<接口字段、错误码、数据结构等>
```

设计段结构化、信息密集、不放图、不用 mermaid（模块图用文字树或表格）。

收尾：

> ✅ 技术设计已落档：`.cadence/cycle-<主题>/may-<主题>.md`，源需求文档未动，may 文档未复制 pai 正文。复核后交下游实施。

# 调研循环（每轮追问前扫描）

1. 从用户最新回复 + 设计草稿挑出具名的库/框架/协议/标准/第三方服务/SDK/云服务/模型 API/算法。
2. `ls .cadence/research/ 2>/dev/null`，文件名匹配的视为已覆盖、不再调研；若该主题的事实要写进设计段，定点 Read 这一份再 inline。
3. 剩余候选判断**会否阻塞下游**：现在不查，下游做计划或写代码时会因信息缺口卡住吗？外部服务/SDK 的接口路径/鉴权/请求响应结构/错误码/限流/模型 ID、版本敏感的 breaking change、协议/合规字段约束、影响代码结构的选型、有非通用细节的算法——任一命中即触发。常识补充、用户已说清、通用编程模式不触发。
4. 触发 → AskUserQuestion 问「调研」还是「跳过」。用户同意：

```bash
mkdir -p .cadence/research
```
```
Agent(subagent_type="cadence:research-agent", description="调研 <topic>",
      prompt="[topic] <一句话主题>\n[output_dir] .cadence/research\n按 research-agent 系统约定产出调研笔记，落到 output_dir 下。")
```

完成后 Read 产物作后续上下文；失败则告知"调研未成功，跳过"继续。

# 异常

- 不像开发任务（闲聊/求助）→ 说明 may 用途，不强行进流程。
- 中途说"算了" → 不写文件，礼貌退出，源 pai 不动。
- 参数缺失 → 见阶段 0。Bash/Write 失败 → 报错不重试。
