# /cadence:spec 设计（已定稿）

## 定位
整个 cycle 的入口。**唯一目标：让用户和模型对"这次要做什么"达成共识**，落到 `REQUIREMENT.md` 和 `REQUIREMENT.html`。

明确不做：
- 不读代码
- 不谈技术方案、技术栈、性能要求
- 不谈风险与不确定项
- 不拆任务

## 调用形式
```
/cadence:spec
/cadence:spec <一句话描述>     # 可选起点
```

## 启动前置检查（必须最先做，顺序执行）

### 1. 检测当前是否有未结 cycle

读取 `.cadence/CURRENT`：

- 文件不存在 / 内容为空 → 通过，继续
- 文件存在且内容非空 → **直接报错退出**。提示用户**两条出路**：
  - 运行 `/cadence:archive` 收尾上一个 cycle（正常归档）
  - 运行 `/cadence:cleanup` 放弃上一个 cycle（中途不想做了）

不允许在已有 cycle 未结的情况下开始新的 spec —— spec 命令必须串行执行。

### 2. 读取项目档案 PROJECT.md

读取 `.cadence/PROJECT.md`：

- 不存在 → 静默跳过（首次使用 cadence，正常情况）
- 存在 → Read 完整内容，作为后续追问与"特性是否已存在"判断的上下文

**只读不写**。spec 命令任何环节都不修改 PROJECT.md（PROJECT.md 由 archive 维护）。

### 3. 特性已存在的识别

结合用户初始输入与 PROJECT.md 内容，判断本次需求是否疑似已被某个已记录 cycle 实现 / 覆盖。

判定为"疑似已存在"时，用 **AskUserQuestion** 让用户三选一：

- **继续做新需求**（确认是不同于现有的新东西）→ 进入正常追问流程
- **改用 design 阶段改造已有**（其实是在已有基础上扩展）→ 提示用户运行 `/cadence:design`，本命令退出
- **取消，我去看看现有的** → 退出

判定不命中 / PROJECT.md 不存在 → 静默进入正常追问流程，不打扰用户。

## 硬规则

- **全程中文输出**。
- **不读项目源代码**。spec 阶段只聊需求，不看代码。
- **不谈技术方案 / 技术栈 / 性能 / 风险 / 不确定项**。这些留给 `/cadence:design`。
- **不拆任务**。这是 `/cadence:run` 的事。
- **每轮最多追问 1~2 个问题**，不连珠炮。
- **没有快速通过出口**：用户即使说"别问了直接落档"，只要内部 gate 没走完就继续问，确保需求质量。"用户已经不耐烦了"不是放行条件。

## 追问维度（只聊纯需求）

1. 目标用户 / 使用场景
2. 核心价值 / 要解决的问题
3. **范围边界（做什么 + 不做什么）** —— 一等公民，自检单独 gate
4. 验收标准

技术栈、性能、约束、风险全部留给 `/cadence:design`。

## 范围边界的特别要求

- 对每个"做什么"，必须反问一个对应的"不做什么"
- 对模糊词汇必须追问消歧（例："登录" → 第三方？记住我？忘记密码？）
- 主动列举"用户没说但通常会期待"的功能，逐一确认是否在范围内
- 自检时单独问："只看这份范围，开发者会不会画蛇添足或漏做？"
- 此项不通过则**绝不进入**"是否还想深入聊"的环节

## 内部自检逻辑

每轮对话后**完全静默**地内部判断（不向用户输出自检状态、不打勾打叉、不报告进度）：

- [范围检查]：所有"做"都有对应"不做"？所有模糊词都消歧了？常见隐性期待都问过了？
- [其他维度检查]：目标用户 / 核心价值 / 验收标准都清楚了？
- 把自己当作下一阶段的设计执行者：信息够开工了吗？

任一不通过 → 继续追问。两个检查都通过 → 进入"留口"。

## 调研建议机制

spec 进行中，主 agent 识别到下列情况时**主动询问用户**是否调研：

- 涉及陌生业务领域
- 涉及合规 / 法规 / 标准
- 用户希望参考竞品但未给出具体参考

不触发：在已有需求基础上的常识性补充、用户已说清楚的事项。

询问用 **AskUserQuestion**，两个选项：

- **继续调研**
- **跳过**

### 用户同意调研时的流程

调研环节会**提前确定 slug 并创建 cycle 目录**（不等到落档环节）：

1. 根据已澄清的需求总结一个简短英文 kebab-case slug（如 `add-login`、`mvp-blog-site`、`refactor-auth`）
2. **slug 一旦定下，后续即使需求方向变化也不再改名**
3. Bash 执行 `mkdir -p .cadence/cycle-<slug>/research`
4. 用 Agent 工具调起 `research-agent`，产物落到 `.cadence/cycle-<slug>/research/<topic-slug>.md`
5. 调研完成后 Read 产物，作为后续追问的上下文
6. research-agent 失败 → 告知用户"调研未成功，跳过"，继续主流程

用户拒绝 → 不调，继续追问。slug 与目录留到落档环节再处理。

详见 05-subagents.md。

## 留口环节

两个内部检查都通过后，用 **AskUserQuestion** 询问用户，两个选项：

- **直接落档**
- **还想再聊**（用户选这个会回到追问循环，下一轮 gate 通过后再次出现本环节）

**用 AskUserQuestion 而非自由文本**，确保用户决策被结构化捕捉，避免模型把用户的犹豫语气误判为"想继续"或"想结束"。

## 落档环节

用户在留口环节选"直接落档"后执行：

### Step 1：定 slug

- 调研环节已经定过 slug → 直接复用，跳过 Step 2 的 mkdir
- 否则根据本次需求总结一个简短英文 kebab-case slug。**不让用户确认、不让用户改**。

### Step 2：建目录

Bash 执行 `mkdir -p .cadence/cycle-<slug>`。已存在则**直接覆盖**，不前置判断、不报错。

### Step 3：写 CURRENT 游标

用 Write 工具写 `.cadence/CURRENT`，内容为 `cycle-<slug>`。

### Step 4：分别写 REQUIREMENT.md 与 REQUIREMENT.html

**两份内容不一样**，是同一需求的两种表达，用 Write 工具分别写入。

### Step 5：收尾

提示下一步 `/cadence:design`。

## 产物：REQUIREMENT.md（给模型读）
- 结构化、信息密集、无装饰
- 用清单、键值对、"做/不做"对照
- 验收标准写成可勾选项，措辞严格（"用户能..."而非"支持..."）
- 不省略任何边界条件，宁可重复也要写清
- 不放图

骨架（不强制套）：
```markdown
# <自然语言标题>

<段落叙述：要做什么、给谁用、解决什么>

## 做什么
- ...

## 不做什么
- ...

## 验收
- [ ] ...
```

## 产物：REQUIREMENT.html（给用户读）
- 视觉化、图文并茂、扫读高效
- 整段叙述讲清"为什么做、给谁、要达成什么"
- Mermaid 流程图展示用户旅程（用 CDN 引入）
- 卡片式 HTML/CSS 展示"做什么 vs 不做什么"对比
- 验收标准用可视化进度卡或图标列表
- 可省略 md 里为消歧而写的细枝末节

## 错误兜底

- 用户初始输入完全不像"开发任务"（聊天闲聊、技术求助等）→ 礼貌说明 spec 命令的用途，不强行进入流程
- 用户中途说"算了不做了" → 不写任何文件，礼貌退出
- **中途取消时**：如果调研环节已经创建了 `.cadence/cycle-<slug>/` 目录，**不主动清理**——清理由专门的 `/cadence:cleanup` 命令负责（spec 命令只关心 spec 阶段的产物，不管目录残留）

## 关键约束（用户明确表达过的偏好）
- 启动前置检查必须最先做：CURRENT 非空就拦，提示 archive 与 cleanup 两条出路
- PROJECT.md 只读不写
- 特性已存在的识别用 AskUserQuestion 三选一
- 没有"用户不耐烦"快速通过出口
- 调研环节提前定 slug 并建目录，slug 一旦定下不再改
- 追问节奏：每轮 1~2 个，不连珠炮
- 不让用户确认 slug
- 不前置判断目录是否存在，直接覆盖
- 自检通过后必须留"还想深入聊吗"的开口给用户，且用 AskUserQuestion 而非自由文本
- md 和 html 内容**不应一样**，是同一需求的两种表达
- 中途取消时的目录残留由 `/cadence:cleanup` 处理，spec 不管
