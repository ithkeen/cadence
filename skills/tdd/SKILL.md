# 测试驱动开发

## 理念

**核心原则**：测试应当透过公共接口去验证行为，而不是验证实现细节。代码可以推倒重写，测试不该跟着动。

**好测试**是集成风格的：透过公共 API 走真实的代码路径，描述的是系统*做什么*，而不是*怎么做*。一个好测试读起来像一份规约——"用户能用有效购物车结账"准确告诉你存在什么能力。这类测试在重构中能活下来，因为它们不关心内部结构。

**坏测试**则与实现耦合：mock 内部协作者、测试私有方法、或者绕过接口走外部手段去验证（比如直接查数据库而不是走接口）。预警信号是：你重构时测试挂了，但行为没变。如果只是改了一个内部函数的名字测试就挂，那这些测试测的是实现，不是行为。

示例见 [tests.md](tests.md)，mock 指南见 [mocking.md](mocking.md)。

## 反模式：水平切片

**不要先写完所有测试再写所有实现。** 这就是"水平切片"——把 RED 理解成"写所有测试"，把 GREEN 理解成"写所有代码"。

这样做会写出**垃圾测试**：

- 批量写出来的测试验证的是*想象中*的行为，不是*真实*的行为
- 你最后测的是东西的*形状*（数据结构、函数签名），而不是用户能看到的行为
- 测试对真实变化变得不敏感——行为坏了它通过，行为没坏它失败
- 你超出了自己的视野范围——在还没理解实现之前，就把测试结构锁死了

**正确做法**：使用"曳光弹"做垂直切片。一个测试 → 一个实现 → 重复。每个测试都是对上一轮所学的回应。代码刚刚是你自己写的，所以你准确知道哪些行为重要、该怎么验证。

```
错（水平）：
  RED：   test1, test2, test3, test4, test5
  GREEN： impl1, impl2, impl3, impl4, impl5

对（垂直）：
  RED→GREEN： test1→impl1
  RED→GREEN： test2→impl2
  RED→GREEN： test3→impl3
  ...
```

## 工作流

### 1. 规划

探索代码库时，使用项目的领域术语表，让测试名和接口词汇与项目本身的语言保持一致；同时尊重你所触及区域的 ADR。

动手写代码之前：

- [ ] 与用户确认需要哪些接口变更
- [ ] 与用户确认要测哪些行为（排好优先级）
- [ ] 识别可以做成[深模块](deep-modules.md)的地方（小接口、深实现）
- [ ] 按[可测试性](interface-design.md)设计接口
- [ ] 列出要测的行为（不是实现步骤）
- [ ] 让用户批准这份计划

问一句："公共接口应该长什么样？哪些行为最值得测？"

**你不可能把所有东西都测了。** 与用户确认到底哪些行为最关键。把测试精力集中在关键路径和复杂逻辑上，不要把每一个可能的边界情况都覆盖。

### 2. 曳光弹

只写一个能验证系统一件事的测试：

```
RED：   为第一个行为写测试 → 测试失败
GREEN： 写最少的代码让它通过 → 测试通过
```

这就是你的曳光弹——证明这条路能端到端跑通。

### 3. 增量循环

对剩下的每一个行为：

```
RED：   写下一个测试 → 失败
GREEN： 写最少的代码让它过 → 通过
```

规则：

- 一次一个测试
- 只写够让当前测试通过的代码
- 不要预判后续测试
- 测试只盯可观察的行为

### 4. 重构

所有测试通过后，找[重构候选](refactoring.md)：

- [ ] 抽出重复代码
- [ ] 把模块加深（让复杂度藏到简洁的接口背后）
- [ ] 在自然的地方应用 SOLID 原则
- [ ] 想一想新代码让你对既有代码有了什么新认识
- [ ] 每一步重构之后都跑一遍测试

**红灯状态下绝不重构。** 先回到绿灯再说。

## 每轮循环的检查清单

```
[ ] 测试描述的是行为，不是实现
[ ] 测试只走公共接口
[ ] 内部重构后测试仍能存活
[ ] 这一轮写出来的代码是最小化的
[ ] 没有加入投机性的功能
```
