# cadence:test-case-design

# 这个流程做什么

输入 `cadence:pai` 产出的需求文档，根据其中的 `REQ`、`AC`、边界异常和相关上下文，设计一份行为测试用例文档。只设计测试用例，不写测试代码，不改需求文档。

# 全局约束

- 全程中文。pai 文档是既定需求来源；不要重写需求，不要补造业务规则。
- 测试用例测功能，不测实现细节。优先通过 API、CLI、页面交互、对外方法或通用函数等公共入口验证用户可观察行为。
- 不为内部实现逻辑、内部 helper、私有状态、框架机制或只为覆盖率存在的分支单独设计用例。
- 用例必须少而精：覆盖关键路径、业务规则、错误处理和高风险分支，不追求穷举；每个用例都要能回答“它验证了哪种功能场景”。
- Mock 只放在系统边界：外部 API、时间 / 随机数、必要的文件系统边界。数据库优先用测试数据库；只有没有测试数据库时才写 mock 数据库，并标明风险。

# 流程

## 0. 确认 pai 文档路径 + 定位

进入 `cadence:test-case-design` 后，先确认本轮要处理的 pai 需求文档路径。若触发文本里已经明确给出路径，直接使用；否则只问路径，等用户回复后再继续。

拿到路径后记为 `<pai-path>`；路径不可读 → 告诉用户 `❌ 路径不可读：<pai-path>`，让用户重新给路径，不继续设计。

**主题**：取 `<pai-path>` 文件名去 `.md`，若以 `pai-` 开头则剥掉前缀。**产物路径**：`<pai 同目录>/test-cases-<主题>.md`。

产物已存在 → 问用户「覆盖重做」还是「退出」。

## 1. 加载 pai 文档

读取 `<pai-path>` 全文。按以下信息抽取测试设计输入：

- 每个 `REQ` 的类型、用户 / 调用方、触发条件、输入、期望行为、输出、边界与异常、不包含。
- 每条 `AC` 的可观察结果。
- `相关上下文` 里的现状定位、已知事实、设计前必查和待核验问题。
- 全局范围外、全局约束和术语表。

如果 pai 路径不可读，报错并停止，不写测试用例文档。

## 2. 设计测试用例

对每个 `REQ` 先确定公共测试入口；入口不明确时写成 `待确认：<缺口>`，不要改测内部函数。

按以下优先级选用例：

1. 覆盖每条 `AC` 的最小用例集合。
2. 覆盖关键边界异常和失败路径。
3. 覆盖全局约束中明确要求验证的兼容性、权限、安全、性能或迁移行为。

不要为同一个行为拆出多个只差实现步骤的用例。发现需求缺口时，写入 `待确认问题`，不要向用户追问后再落档。

## 3. 落档

写入 `<pai 同目录>/test-cases-<主题>.md`，一次写入：

```markdown
# 测试用例设计：<主题>

## 需求来源
- PAI 文档：`<pai-path>`
- 覆盖范围：REQ-001、REQ-002

## 测试策略
- 测试层级：<接口 / 集成 / 端到端 / 单元；按需求选择>
- 公共入口：<API / CLI / 页面 / 对外方法 / 通用函数；未知则写待确认>
- 数据策略：<测试数据库 / fixture / 构造数据 / 不需要数据>
- Mock 策略：<无 / 外部 API / 时间 / 文件系统 / 数据库；数据库 mock 必须说明原因>
- 不测项：<内部实现、范围外需求、已明确不包含的行为>

## 需求到用例映射

| REQ | AC | 用例 |
|---|---|---|
| REQ-001 | AC-001 | TC-001 |

## 测试用例

| ID | 来源 | 测试入口 | 行为 | 场景 / 前置条件 | 操作 | 预期结果 | Mock / 数据策略 | 优先级 |
|---|---|---|---|---|---|---|---|---|
| TC-001 | REQ-001 / AC-001 | <公共入口> | <验证什么功能> | <关键条件> | <通过公共入口执行什么> | <用户可观察结果> | <无 / 外部 API / 测试数据库等> | P0 |

## 待确认问题
- <影响测试设计但 pai 未确认的信息；没有则写“无”>

## 质量检查
- 每条 AC 至少映射到一个测试用例，或在待确认问题中说明为什么暂不能映射。
- 每个用例都通过公共入口验证可观察行为。
- 没有测试内部函数、私有状态、调用顺序或框架机制。
- Mock 只出现在系统边界；数据库优先用测试数据库。
- 用例数量足够少，删除任一 P0/P1 用例都会少覆盖一个重要行为。
```

收尾：

> ✅ 测试用例设计已落档：`.cadence/cycle-<主题>/test-cases-<主题>.md`，源 pai 文档未动。

# 异常

- 不像 pai 需求文档 → 说明本流程只消费 `cadence:pai` 产物，不强行设计。
- 中途说"算了" → 不写文件，礼貌退出，源 pai 不动。
- pai 文档路径缺失或不可读 → 见阶段 0，先让用户补路径或修正路径。读取 / 文件写入失败 → 报错不重试。
